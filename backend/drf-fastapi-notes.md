# DRF + FastAPI 설계 노트

## 왜 둘 다 쓰는가
- DRF: 계정/권한/관리형 CRUD, 운영 안정성이 중요한 도메인
- FastAPI: 실시간 처리, AI 파이프라인, 경량 API

핵심은 프레임워크 선택보다 "경계 명확화"다.

## 인터페이스 설계 원칙
1. 요청/응답 스키마를 먼저 문서화한다
2. 에러 코드를 기능별로 통일한다
3. FE/HW/AI가 같은 필드 이름을 쓰게 한다

## API 명세 체크리스트
- 엔드포인트 목적이 1문장으로 설명 가능한가
- 필수/선택 필드가 명확한가
- 실패 케이스와 응답 코드가 정의되어 있는가
- 타임아웃/재시도 전략이 문서에 있는가

## 에러 코드 예시 (개인 기준)
| 코드 | 의미 | 처리 방식 |
|------|------|-----------|
| 400 | 입력값 오류 | 사용자 입력 재검증 |
| 401 | 인증 실패 | 재로그인/토큰 갱신 |
| 403 | 권한 없음 | 접근 차단/권한 안내 |
| 404 | 리소스 없음 | 대상 유효성 재확인 |
| 409 | 상태 충돌 | 동시성/중복 요청 처리 |
| 500 | 서버 오류 | 로그 추적 후 재시도 정책 적용 |

## 회고
- PM 전환 이후 가장 큰 변화는 "개발 속도"가 아니라 "재작업 감소"였다
- 문서화와 규약 정리가 실제 일정 리스크를 줄였다
