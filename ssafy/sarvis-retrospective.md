# SARVIS 회고: FE에서 PM으로 전환하며 배운 것

## 프로젝트 정보
- 기간: 2026.01.07 ~ 2026.02.09
- 인원: 6명
- 역할: FE 시작 -> 스프린트 2 시작 시점 PM 전환

## 전환 배경
- 스프린트 1 종료 시점에 FE/BE/HW 간 충돌이 반복됨
- 같은 기능을 각자 다른 기준으로 구현하고 있었음
- 핵심 문제는 개발 속도가 아니라 "공통 기준 부재"였음

## 내가 한 조치
1. 통신 구조 검토 및 누락 규약 정의
2. API 상세 명세 작성 강제
3. 작성한 명세를 FE/HW에 동기 배포
4. 변경사항은 문서 기준으로 의사결정

## 실수와 개선
- 실수: PM이 기획만 하면 된다고 생각함
- 개선: PM은 구현 가능한 상세 기준까지 책임져야 함
- 다음에 같은 상황이라면:
  - 초기 명세/설계 먼저 확정
  - 디자인 리소스 이탈 리스크를 초기에 관리

## 면접에서 말할 포인트
- 갈등을 "사람 문제"가 아니라 "기준 문제"로 바꿔 해결했다
- 시스템 전체 관점으로 문제를 재정의해 재작업 비용을 줄였다
